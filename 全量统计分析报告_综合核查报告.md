## 一、致命缺陷（导致数据完全不可用）

### 缺陷 #1 — `_normalize_results` 数据适配层完全失效

**位置**: `core/core/seat_evaluation/report_builder.py` 第 184-310 行

**问题描述**: `_normalize_results()` 函数从 `evaluator_results.get('metrics', {})` 中查找 `location_results` 子键：

```python
for loc_id, loc_data in metrics.get('location_results', {}).items():
```

但 `FullTimeseriesEvaluator.results['metrics']` 是 `comprehensive_metrics()` 返回的**扁平字典**（如 `{'exp_Ax_VDV': ..., 'exp_Ay_RMS': ...}`），不存在 `location_results` 键。循环永不执行。

**后果**:

| 报告节 | 实际填充内容 | 后果 |
|--------|-------------|------|
| `time_domain.vdv` | **空 `{}`** | VDV全部为0 |
| `time_domain.acc_peak` | **空 `{}`** | 峰值加速度全部为0 |
| `time_domain.gyro` | **空 `{}`** | 角速度全部为0 |
| `frequency_domain.seat` | **空 `{}`** | SEAT因子全部为0 |
| `frequency_domain.transfer` | **空 `{}`** | 传递系数全部为0 |
| `shock_fatigue.iso2631_5` | **空 `{}`** | S_d全部为0 |

**影响范围**: 所有下游模块（舒适度指数、主观评价、脊柱健康、平顺性、调校建议、趋势监测）全部基于**全零数据**计算，产生虚假高分。

**修复方案**: `_normalize_results` 应从 `report['location_results']` 而非 `evaluator_results['metrics']` 提取指标，或重构数据流使 `evaluator_results` 包含 `location_results` 结构。

### 缺陷 #2 — `S_d` 数据路径错误（3个模块同时出现）

**位置**:
- `core/core/seat_evaluation/comfort_index.py` 第 95 行
- `core/core/seat_evaluation/seat_tuning_advisor.py` 第 131-133 行
- `core/core/seat_evaluation/trend_monitor.py` 第 74 行

**问题描述**: 三处代码均写入：

```python
sf.get('iso2631_5', {}).get('实验组', {}).get('S_d', 0)
```

但 `_normalize_results` 构建的实际结构是：

```python
iso2631_5 = {
    '座垫': {'实验组': {'S_d': ...}, '对照组': {'S_d': ...}},
    '靠背': {'实验组': {'S_d': ...}, '对照组': {'S_d': ...}},
    ...
}
```

缺少了 `座垫` 这一层，`.get('实验组', {})` 永远返回 `{}`，`S_d` 永远为 0。

**影响**: 即使缺陷 #1 修复后，此缺陷也会导致冲击得分恒为满分（100分），冲击相关分析完全失效。

**修复方案**: 统一为 `iso2631_5[loc_name]['实验组']['S_d']`，添加 `_get_first_location_sd()` 辅助函数自动获取第一个位置的 S_d 值。

### 缺陷 #3 — 频段定义重复，导致 `attenuation.overall` 双倍计数

**位置**: `core/core/seat_evaluation/full_timeseries_evaluator.py` 第 39-50 行

**问题描述**: `freq_bands` 配置中，同一频段定义了两次（英语名 + Hz名）：

```python
'freq_bands': {
    'sub_low': (0.1, 0.5),     # 与 0.1-0.5Hz 重复
    'low': (0.5, 1.0),          # 与 0.5-1Hz 重复
    'mid_low': (1.0, 5.0),      # 与 1-5Hz 重复
    'mid_high': (5.0, 20.0),    # 与 5-20Hz 重复
    'high': (20.0, 80.0),       # 与 20-80Hz 重复
    '0.1-0.5Hz': (0.1, 0.5),
    '0.5-1Hz': (0.5, 1.0),
    '1-5Hz': (1.0, 5.0),
    '5-20Hz': (5.0, 20.0),
    '20-80Hz': (20.0, 80.0),
},
```

`bands_atten` 包含 **10个条目**（5对重复），`np.mean()` 对其求均值时，每个频段被双倍计数，`overall` 值不准确。

**修复方案**: 删除 `freq_bands` 中重复的英文键名（`sub_low`、`low`、`mid_low`、`mid_high`、`high`），只保留 Hz 格式的 5 个频段。

### 缺陷 #4 — `subjective_mapping.py` 中 XY 轴贡献被重复累加

**位置**: `core/core/seat_evaluation/subjective_mapping.py` 第 33 行

**问题描述**:

```python
aw_total = np.sqrt((1.4 * aw_z) ** 2 + aw_xy ** 2 + aw_xy ** 2)
```

`aw_xy ** 2` 被加了**两次**，等效于 `aw_xy ** 2 * 2`。正确的公式应为：

```python
aw_total = np.sqrt((1.4 * aw_z) ** 2 + aw_xy ** 2)
```

**影响**: `aw_total` 被等比例放大（系数约 √2 ≈ 1.41），导致不适等级判定偏高 1-2 级（例如本应是"轻微不适"被判定为"中等不适"）。

**修复方案**: 删除重复的 `+ aw_xy ** 2`。

## 二、严重问题

### 问题 #5 — VDV → aw 转换粗糙且不一致

**位置**:
- `core/core/seat_evaluation/report_builder.py` 第 101-102 行：`aw_z = VDV_Z / 10`
- `core/core/seat_evaluation/ride_quality.py` 第 102-104 行：`aw = VDV / T^(1/4)` (T=600)

**问题描述**: 同一 VDV→aw 转换，两个模块用了**不同公式**：

| 模块 | 公式 | 示例（VDV=10） |
|------|------|---------------|
| `report_builder.py`（主观评价） | `aw = VDV / 10` | aw = 1.0 |
| `ride_quality.py`（平顺性） | `aw = VDV / 600^0.25` | aw ≈ 2.02 |

两者相差约 **2倍**。且 VDV→aw 的近似关系 `VDV ≈ a_w * T^(1/4)` 仅适用于**稳态随机振动**，不适用于包含冲击的驾驶工况。

**修复方案**: 统一 VDV→aw 转换公式，或直接使用加速度 RMS 而非 VDV 近似；建议从原始数据中直接计算频率加权加速度 RMS。

---

### 问题 #6 — `spine_health.py` 将 VDV 直接当作 D_k 使用

**位置**: `core/core/seat_evaluation/spine_health.py` 第 104-106 行

**问题描述**: 代码注释明确写明了转换关系（`D_k ≈ VDV * T^(1/12)`），但实际代码未执行转换，直接将 VDV 值赋给 D_k：

```python
sr.d_x = seat_vdv.get('X', 0)  # 应该是 VDV * T^(1/12)
sr.d_y = seat_vdv.get('Y', 0)  # 应该是 VDV * T^(1/12)
sr.d_z = seat_vdv.get('Z', 0)  # 应该是 VDV * T^(1/12)
```

**影响**: VDV 量纲为 m/s^1.75，D_k 量纲应为 m/s^2。缺少 `T^(1/12)` 转换因子（对于 T=600s，约 1.65 倍），导致 S_e 和风险因子 R 的值偏低，风险等级被低估。

**修复方案**: 添加 `D_k = VDV * T^(1/12)` 转换，或从原始加速度数据中按 ISO 2631-5 Annex C 正确计算加速度剂量。

### 问题 #7 — `comfort_index.py` 的 `transfer_score` 无下限保护

**位置**: `core/core/seat_evaluation/comfort_index.py` 第 100-102 行

**问题描述**:

```python
ci.transfer_score = min(100, avg_atten * 2)
```

当衰减率为负值（座椅放大振动，即 `avg_atten < 0`）时，`avg_atten * 2` 为负，`min(100, 负数)` 返回负数。得分应为 0-100 范围，缺少 `max(0, ...)` 下限保护。

**修复方案**: 改为 `ci.transfer_score = max(0, min(100, avg_atten * 2))`。

### 问题 #8 — `attenuation.overall` 是频段均值的均值，非整体衰减率

**位置**: `core/core/seat_evaluation/report_builder.py` 第 298 行

**问题描述**:

```python
atten[ax] = {'overall': np.mean(list(bands.values()))}
```

`overall` 被定义为各频段衰减率的算术平均，而非物理意义上的"整体衰减率"。

- **当前定义**: `overall = mean(各频段衰减率)`
- **正确物理定义**: `overall = 1 - exp_RMS_total / ctrl_RMS_total`

两个指标含义不同，报告中标注为"衰减率(%)"会产生误导。

**修复方案**: 将 `overall` 改为从 RMS 比值计算的整体衰减率，或将字段重命名为 `band_mean_attenuation` 以明确语义。

### 问题 #11 — `comfort_index.py` 的 `S_d` 阈值与 `spine_health.py` 的 `R` 阈值语义矛盾

**位置**:
- `core/core/seat_evaluation/comfort_index.py` 第 66-68 行
- `core/core/seat_evaluation/spine_health.py` 第 51-55 行

**问题描述**:

| 模块 | 指标 | 阈值体系 |
|------|------|---------|
| `comfort_index.py` | S_d (MPa) | ≤0.3→100分, ≥1.2→20分 |
| `spine_health.py` | R = S_e/S_ut | ≤0.5→低风险, ≥0.8→高风险 |

S_d 和 R 并非同一指标：S_d 是压缩应力 (MPa)，R = S_e / S_ut 是风险因子。两者阈值体系和语义不同，但在报告中相邻出现（"冲击得分" vs "脊柱风险"），可能造成用户困惑：冲击得分高但脊柱风险也高？需要有明确的关联说明。

**修复方案**: 在报告中对两个指标的定义和关联关系添加说明文字，或统一使用同一套 ISO 2631-5 评价体系。

### 问题 #13 — `seat_tuning_advisor.py` 的 `PSD_Z_peak` 提取逻辑不健壮

**位置**: `core/core/seat_evaluation/seat_tuning_advisor.py` 第 136-145 行

**问题描述**:

```python
psd_z = psd.get('Az', {})
if psd_z:
    if isinstance(psd_z, dict):
        max_freq = max(psd_z.keys(), key=lambda k: float(psd_z[k]))
```

PSD 字典的键是字符串（来自 `_normalize_results` 的 `str(k)` 转换），`float(psd_z[k])` 转换的是 PSD 值而非频率。`max()` 选出的是**PSD值最大**对应的键，而非频率最大。变量名 `max_freq` 和实际语义不符——实际选出的是**共振峰所在频率**（PSD峰值对应的频率），变量名应改为 `resonance_freq` 以反映真实含义。

**修复方案**: 将变量名改为 `resonance_freq` 或 `peak_psd_freq`，确保语义正确。

### 问题 #14 — `multi_condition_comparator.py` 的 `posture_score` 默认值硬编码

**位置**: `core/core/seat_evaluation/multi_condition_comparator.py` 第 119 行

**问题描述**:

```python
c.get('posture_score', 60),
```

雷达图中 `posture_score` 默认值为 60，而其他维度（振动、冲击、传递）默认值为 0。不一致的默认值导致雷达图在数据缺失时形态扭曲——姿态维度看起来总是"中等偏上"，与其他维度的零点基准不一致。

**修复方案**: 统一默认值为 0，或将所有维度的默认值统一为 50（中位值）。

### 问题 #15 — `trend_monitor.py` 退化检测对 `comfort_index` 和 `seat_z` 方向不统一

**位置**: `core/core/seat_evaluation/trend_monitor.py` 第 100-115 行

**问题描述**: 退化检测对所有指标使用同一方向判断（`old_val - new_val > threshold` → 退化）：

| 指标 | 物理含义 | 下降的物理意义 | 当前判定 |
|------|---------|---------------|---------|
| `comfort_index` | 越高越舒适 | 性能退化 | 退化 ✅ |
| `seat_z` | 越低隔振越好 | 性能改善 | 退化 ❌ |
| `vdv_z` | 越低越舒适 | 性能改善 | 退化 ❌ |
| `sd` | 越低冲击越小 | 性能改善 | 退化 ❌ |

`seat_z`、`vdv_z`、`sd` 下降意味着**性能改善**而非退化，当前判定逻辑与物理意义相反。

**修复方案**: 为每个指标设置方向标记（`higher_is_better` / `lower_is_better`），退化检测时根据方向判断。

### 问题 #17 — `report_builder.py` 中 `events` 字段可能被 DataFrame 覆盖

**位置**: `core/core/seat_evaluation/report_builder.py` 第 71-74 行

**问题描述**:

```python
for key in ['events', 'time_domain', 'frequency_domain', ...]:
    if key in analysis_results:
        report[key] = analysis_results[key]
```

`analysis_results['events']` 来自 `behavior_summary.get('events', [])`（事件列表），但 `evaluator_results['events']` 是 `event_analysis()` 返回的 **DataFrame**。如果 `analysis_results` 的构建路径不同，DataFrame 可能覆盖列表，导致 PDF 报告的事件详情表渲染失败。

**修复方案**: 区分事件列表（`event_list`）和事件分析 DataFrame（`event_analysis`），使用不同的键名。

## 四、指标/算子/派生指标矛盾汇总

| 序号 | 矛盾项 | 模块A | 模块B | 冲突描述 |
|------|--------|-------|-------|---------|
| 1 | **VDV→aw 转换** | `report_builder.py`: `aw = VDV / 10` | `ride_quality.py`: `aw = VDV / T^(1/4)` | 同一VDV值得出相差约2倍的aw，平顺性等级判定不一致 |
| 2 | **S_d 访问路径** | `comfort_index.py`: `.实验组.S_d` | `report_builder.py`: `.座垫.实验组.S_d` | 同一数据两种路径，前者永远取不到值，冲击得分恒为满分 |
| 3 | **衰减率定义** | `report_builder.py`: 频段均值平均 | `full_timeseries_evaluator.py`: 整体RMS比 | "衰减率"指标在两个模块含义不同，报告中的数值与物理意义脱节 |
| 4 | **退化方向** | `trend_monitor.py`: `seat_z`↓=退化 | 物理意义: SEAT↓=改善 | 指标方向与物理意义相反，趋势监测给出错误告警 |
| 5 | **冲击 vs 脊柱** | `comfort_index.py`: S_d阈值 (0.3-1.2 MPa) | `spine_health.py`: R阈值 (0.5-0.8) | 两个"冲击相关"指标使用不同标准体系，无关联说明，用户无法理解两者关系 |
